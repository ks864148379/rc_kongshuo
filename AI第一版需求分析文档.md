内部 HTTP 通知投递服务 — 技术设计方案

一、背景与目标

1.1 业务背景

企业内部多个业务系统在关键事件发生时，需要调用外部供应商的 HTTPS API 进行通知：
- 用户通过第三方广告系统引流注册成功 → 通知广告系统
- 用户订阅付款成功 → 通知 CRM 系统变更 Contact 状态
- 用户购买商品 → 通知库存系统进行库存变更

1.2 核心挑战

- 不同供应商 API 的请求地址、Header、Body 格式各不相同
- 业务系统本身不关心外部 API 返回值
- 需要确保通知请求能被稳定、可靠地送达

1.3 设计目标

┌────────┬─────────────────────────────────────────────────────────┐
│  目标  │                          说明                           │
├────────┼─────────────────────────────────────────────────────────┤
│ 可靠性 │ At-least-once 投递语义，通知接收后绝不丢失              │
├────────┼─────────────────────────────────────────────────────────┤
│ 解耦   │ 业务系统提交即忘（fire-and-forget），由中继服务负责重试 │
├────────┼─────────────────────────────────────────────────────────┤
│ 灵活性 │ 支持任意 URL/Header/Body 组合，适配不同供应商           │
├────────┼─────────────────────────────────────────────────────────┤
│ 可观测 │ 完整的投递状态、重试历史、失败原因可追溯                │
├────────┼─────────────────────────────────────────────────────────┤
│ 可扩展 │ 支持水平扩容以应对高并发                                │
└────────┴─────────────────────────────────────────────────────────┘

1.4 非目标

- 不需要将供应商的响应内容回传给调用方
- 不做请求内容转换/映射（调用方直接提供完整的请求报文）
- 不保证投递顺序（各通知事件相互独立）

核心数据流

1. 业务系统调用 POST /api/v1/notifications 提交通知请求
2. 接入层校验参数、持久化到 notification_task 表（status=PENDING），立即返回 202
3. 调度器轮询就绪任务，使用 SELECT ... FOR UPDATE SKIP LOCKED 加锁获取
4. Worker 线程发送 HTTP 请求到目标地址
5. 收到 2xx → 标记 DELIVERED
6. 收到可重试失败 → 累加重试次数，计算下次重试时间，标记 RETRYABLE
7. 重试次数耗尽 → 移入死信队列

为什么选择数据库队列而非消息中间件（Kafka/RabbitMQ）？

- 运维简单：减少外部依赖，不需要额外维护消息中间件集群
- 事务一致：接收+持久化在同一事务中，不存在"消息发了但没入库"的问题
- 队列语义：MySQL 8.0 的 FOR UPDATE SKIP LOCKED 提供了无竞争的队列消费能力
- 吞吐量足够：通知中继场景下，数千 TPS 的 DB 轮询方案完全满足需求
- 可演进：未来如果吞吐量瓶颈出现，可以在不改变 API 的前提下迁移到消息中间件

六、重试策略

6.1 指数退避 + 随机抖动

delay = min(initialDelay × 2^(attempt-1) + random_jitter, maxDelay)

其中：
initialDelay = 1s（可按任务配置）
multiplier = 2
jitter = delay × random(0, 0.2)  // 防止惊群
maxDelay = 1 小时（上限）

七、调度器设计

7.1 轮询 SQL（MySQL 8.0）

SELECT * FROM notification_task
WHERE status IN ('PENDING', 'RETRYABLE')
AND next_attempt_at <= NOW()
ORDER BY next_attempt_at ASC
LIMIT 50
FOR UPDATE SKIP LOCKED;

- FOR UPDATE SKIP LOCKED：多个服务实例同时轮询时不会重复获取同一条任务
- 批次大小：默认 50，可配置
- 轮询间隔：500ms，可配置

7.2 并发模型

- 调度线程每 500ms 轮询一次数据库
- 获取到的任务分发到有界线程池（默认 20 线程）
- 每个线程独立执行 HTTP 投递
- 投递完成后在事务中更新任务状态

7.3 水平扩展

多个服务实例可以同时运行，SKIP LOCKED 提供了天然的无竞争分区能力，不需要额外的协调机制。

 ---
八、幂等设计

- 业务系统提交时可选填 idempotencyKey
- 数据库层面通过 UNIQUE(source_system, idempotency_key) 约束保证
- 重复提交时返回 409，并附带已有任务的状态
- 防止因调用方自身重试导致的重复投递

10.2 告警规则

- 队列积压 > 10000 持续 5 分钟 → 告警
- 死信率 > 10 条/分钟 → 告警
- P95 投递延迟 > 30s → 告警
- 最老未投递任务 > 1 小时 → 告警

10.3 结构化日志

每条日志包含：taskUid、sourceSystem、eventType、targetUrl（脱敏）、attemptCount、durationMs。

十一、数据清理

- 已投递任务：每日定时清理 N 天前的 DELIVERED 记录（默认 30 天）
- 尝试记录：随父任务一起清理
- 死信：长期保留（支持人工重发），可按策略归档
