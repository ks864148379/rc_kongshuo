# 自主需求分析
需求一句话描述：接收http请求，根据请求内容，把请求路由到不同的下游服务

## 需求边界
- 设定上游传过来的http请求协议符合统一规范，不需要适配不同结构
- 不考虑黑产、风控
- 设定下游需要的body数据都能从上游请求中拿到
- 不考虑上游的重试策略
- 整条链路必须能拿到唯一的幂等id
- 不考虑数据库或者中间件彻底挂掉的情况，允许偶尔的失败
- 不考虑数据库和消息队列性能瓶颈
- 假定调用下游的顺序不依赖上游请求的时序
- 假定调用下游http返回200状态码即成功

## 链路梳理
- Server层： 接收请求 -> 请求关键信息存储到数据库中 -> 根据配置进行请求的转发 -> 更新数据库状态 -> 如果失败把数据写入重试队列
- 重试队列： 获取消息 -> 根据配置进行请求的转发 -> 调用下游 -> 根据状态码更新数据库状态 -> 根据配置的重试时间再次写入重试队列 -> 如果超过重试次数写入失败表
- 定时任务：扫描失败表 -> 发送到重试队列
- 数据库变更消息：接收数据库变更消息 -> 如果最新状态不是已完成，发送到重试队列

整体思路是因为有幂等id的存在 同步转发 + 异步重试 + 定时任务 + 数据库消息 保证消息的发送

# AI第一版方案评审
- AI第一版方案worker对数据库轮询。这里尽量不要轮询数据库
- 这里只用一张表，在一张表中维护请求的状态，不使用死信表和尝试表
- 默认有幂等id的参数设计
- 去掉安全设计、数据清理、可观测性、容量评估部分
- 要求更多考虑请求能尽量转发，设计架构，先不用考虑细节接口实现

# AI第二版方案评审
 - 方案中使用的不是持久化延迟队列，使用的是内存延迟队列。内存延迟队列可能会导致消息丢失，在系统重启的时候需要扫描数据。为了实现方便，采取此方案
 - 方案中没有设计数据库变更消息。此处为了实现方便，不加接数据库变更消息部分
 - 整体技术方案使用JAVA + MySQL，为了能够在不同环境下演示，变更为使用python + prismadb，脚本语言方便演示
